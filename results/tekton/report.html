Integration Surface Level 3 — Optimized
OTLP gRPC/HTTP push for both traces and metrics; Prometheus also supported
▼
What's Working Well
  • OTLP gRPC push confirmed for both traces and metrics
  • OTLP HTTP (http/protobuf) also supported
  • Prometheus endpoint on port 9090 as default
  • Single config-observability ConfigMap controls all telemetry
  • No sidecar or agent required for OTLP push
  • Dedicated migration guide documents the OTel transition
Opportunities
  • Consider making OTLP the default (currently Prometheus) to improve OTel adoption
  • Tracing is disabled by default (tracing-protocol: none) — opt-in reduces visibility
  • The http:// prefix requirement for insecure gRPC is not clearly documented
  • A legacy config-tracing ConfigMap still exists alongside the new one

Configuration Example

✓ metrics-protocol: prometheus → Prometheus scrape (default)
✓ tracing-protocol: grpc → OTLP trace push
⚠ tracing-protocol: none is the default — tracing must be enabled

Key Takeaways

Top Strengths

  1. Full OTLP support for both signals: Tekton supports OTLP gRPC and HTTP push for both traces and metrics via a simple ConfigMap — a significant differentiator from projects that only expose Prometheus.
  2. Rich reconciler traces: 17 span types model the controller lifecycle with proper parent-child hierarchy. Context is propagated across reconciler iterations via K8s annotations — a clever adaptation of W3C TraceContext for the CRD model.
  3. OTel semconv for infrastructure metrics: Go runtime (go.*) and HTTP client (http.client.request.duration) metrics follow current OTel semantic conventions with correct attribute names.
  4. Comprehensive documentation + migration guide: The OpenCensus → OTel transition was documented as a breaking change with detailed before/after tables and action items.
  5. Trace IDs in structured logs: Every log line includes knative.dev/traceid, enabling manual correlation even without OTLP log export.

Priority Improvements

  1. Align service.name across signals: Traces use pipelinerun-reconciler/taskrun-reconciler; metrics use tekton-pipelines-controller. Unifying these would enable automatic cross-signal correlation.
  2. Add attributes to child spans: Only root spans carry pipelinerun/taskrun/namespace attributes. Propagating these to child spans (createPod, resolvePipelineState, etc.) would unlock powerful trace filtering.
  3. Set error status on spans: All spans report UNSET status. Setting ERROR with a message on failed reconciliations would enable trace-based SLO monitoring and alerting.
  4. Add OTLP log export: The trace ID is already in logs — adding a logs-protocol key to config-observability would complete the three-signal story.
  5. Enable OTLP by default: Consider making OTLP the default export path (or at least enabling tracing by default) to improve out-of-the-box observability for OTel-native environments.

Notable Observations

  • Recent OpenCensus → OTel migration (PR #9043): Tekton recently completed a deliberate, well-documented migration. Infrastructure metrics now use OTel semconv naming while preserving backward compatibility for core Tekton metrics.
  • Prometheus endpoint exposes otel_scope_name label: The Prometheus endpoint includes otel_scope_name and otel_scope_schema_url labels — a direct artifact of using the OTel Go SDK's Prometheus bridge. Slightly noisy for PromQL users but shows OTel-native implementation.
  • OTLP gRPC insecure mode requires http:// prefix: The metrics-endpoint must use http:// prefix for insecure connections (e.g., http://collector:4317). Without it, Tekton attempts TLS and fails. This is not clearly documented.
  • service.version is a commit SHA: The version attribute is set to 5a88281 rather than v1.11.1. Operators typically know release versions, not commit hashes.
  • durationAndCountMetrics span: An unusual design — a span specifically tracking when the controller emits metrics. Provides traceability for metric emission but is an implementation detail most operators don't need.
