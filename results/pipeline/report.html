1
Integration Surface
How Tekton connects to the OTel ecosystem
Level 2 · Native

What We Observed

Tekton emits traces via OTLP gRPC directly from controller pods — no sidecar or agent required. Metrics are exposed on a Prometheus endpoint (port 9090) on all four controller components. OTLP gRPC/HTTP is also available for metrics as an opt-in. Logs go to stdout only with no OTLP export path.

Configuration is entirely through Kubernetes ConfigMaps (config-observability), not standard OTEL_* environment variables.

Checklist

  • ✓ OTLP trace export works (gRPC and HTTP/protobuf)
  • ✓ Prometheus metrics endpoint available by default (port 9090)
  • ✓ OTLP metrics export available as opt-in
  • ✓ No sidecar or agent required for telemetry
  • ✗ OTLP log export not available
  • ✗ Tracing is disabled by default (tracing-protocol: none)
  • ✗ No OTEL_EXPORTER_OTLP_ENDPOINT env var support
Opportunity: Adding OTLP log export and enabling tracing by default (even at a low sampling rate like 10%) would complete the three-signal story and bring this to Level 3. Supporting OTEL_EXPORTER_OTLP_ENDPOINT would also enable standard operator tooling like the OTel Operator.

Semantic Conventions
Alignment with OTel standard attribute names and schemas
Level 1 · Aligned

Trace Attributes Observed

Span attributes use custom, non-standard keys:

  • taskrun → should be tekton.taskrun.name or similar namespaced key
  • pipelinerun → should be tekton.pipeline.run.name
  • namespace → should be k8s.namespace.name (OTel semconv)

Schema URL in traces: https://opentelemetry.io/schemas/1.12.0 (2022 — current stable is 1.29.0+)

Instrumentation scope versions: PipelineRunReconciler vunknown, TaskRunReconciler vunknown

What's Working Well

The HTTP auto-instrumentation layer (via go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp v0.67.0) uses current stable semconv correctly:

  • http_request_method, http_response_status_code, server_address, server_port, url_scheme — all correct ✓

Schema URLs are present in both trace and metric exports — showing intentionality even if the versions are stale.

Checklist

  • ✓ Schema URL present in traces and metrics
  • ✓ HTTP layer uses current stable semconv attributes
  • ✗ Span attributes use custom keys instead of OTel conventions
  • ✗ Schema URLs are stale (1.12.0 / 1.18.0 vs current 1.29.0+)
  • ✗ Instrumentation scope version is unknown
  • ✗ No service.version in native resource attributes
Opportunity: Propose a tekton.* namespace in OTel semantic conventions (e.g., tekton.taskrun.name, tekton.pipeline.name). Update schema URLs to 1.29.0 and set instrumentation scope versions to the Tekton release version. This would move Semantic Conventions to Level 2.

Resource Attributes & Configuration
Service identity and operator configurability
Level 1 · Aligned

Native Resource Attributes (before collector enrichment)

Tekton sets only one resource attribute natively:

service.name: "pipelinerun-reconciler"   (or "taskrun-reconciler")

All k8s.* attributes visible in the telemetry files are added by the OTel Collector's k8sattributes processor — not by Tekton itself. Missing natively: service.version, service.namespace, telemetry.sdk.*.

Configuration Interface

All telemetry configuration goes through proprietary Kubernetes ConfigMaps:

  • config-observability: tracing-protocol, tracing-endpoint, metrics-protocol, metrics-endpoint
  • config-tracing: enabled, endpoint, credentialsSecret

Standard OTEL_* environment variables (OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, OTEL_RESOURCE_ATTRIBUTES) are not supported.

Checklist

  • ✓ service.name set natively, meaningfully distinguishing reconciler types
  • ✗ service.version not set natively
  • ✗ No OTEL_* environment variable support
  • ✗ Service identity not consistent across trace and metric signals
  • ✗ telemetry.sdk.* attributes absent from exported data
Opportunity: Add service.version to the OTel resource (set to the Tekton release version). Support OTEL_RESOURCE_ATTRIBUTES to let operators add custom attributes. This would enable the OTel Operator's auto-instrumentation injection pattern and move this dimension to Level 2.

Trace Modeling & Context Propagation
Span hierarchy, naming, and context flow
Level 2 · Native

Span Hierarchy Observed

Tekton produces a well-structured reconciliation span tree:

PipelineRun:Reconciler  [root]
├── updatePipelineRunStatusFromInformer
├── reconcile
│   └── PipelineRun:ReconcileKind
│       ├── resolvePipelineState (×2)
│       ├── runNextSchedulableTask
│       │   └── createTaskRuns → createTaskRun
│       └── updateLabelsAndAnnotations
├── finishReconcileUpdateEmitEvents
└── durationAndCountMetrics

TaskRun:Reconciler  [root]
├── updateTaskRunWithDefaultWorkspaces
├── TaskRun:ReconcileKind
│   ├── prepare → createPod → stopSidecars
└── finishReconcileUpdateEmitEvents

Context Propagation

Internal: SpanContext is persisted in TaskRun/PipelineRun.status.spanContext Kubernetes annotations, allowing the controller to resume a trace across reconciliation loop invocations. Span events confirm this: "updating PipelineRun status with SpanContext".

External: No W3C Trace Context (traceparent/tracestate) ingestion from external callers. A CI system submitting a TaskRun cannot inject a parent span.

Cross-component: PipelineRun and its child TaskRun reconciliations appear as separate traces — no span links connect them.

Checklist

  • ✓ Parent-child span relationships correctly modeled
  • ✓ Span names represent logical operations, not raw code paths
  • ✓ SpanContext persisted in k8s object status for cross-loop continuity
  • ✓ Span events used meaningfully
  • ✗ Root spans are INTERNAL kind — CONSUMER would better represent queue-driven work
  • ✗ No W3C Trace Context ingestion from external callers
  • ✗ PipelineRun and child TaskRun traces not linked
  • ✗ No step-level spans for individual Task step execution
Opportunity: Adding step-level spans (one per Task Step container) would be transformative — enabling operators to see exactly which step in a multi-step pipeline is slow or failing. Linking PipelineRun and TaskRun traces via span links would provide end-to-end pipeline execution visibility.

Multi-Signal Observability
Coverage and correlation across traces, metrics, and logs
Level 2 · Native

Tekton-Native Metrics (11 total)

  • tekton_pipelines_controller_taskrun_duration_seconds — histogram (namespace, status, task)
  • tekton_pipelines_controller_pipelinerun_duration_seconds — histogram (namespace, status, pipeline)
  • tekton_pipelines_controller_pipelinerun_taskrun_duration_seconds — histogram
  • tekton_pipelines_controller_taskrun_total / pipelinerun_total — counters
  • tekton_pipelines_controller_running_taskruns / running_pipelineruns — gauges
  • tekton_pipelines_controller_taskruns_pod_latency_milliseconds — pod scheduling latency
  • Resolution wait gauges: running_pipelineruns_waiting_on_*

Cross-Signal Correlation

Traces and metrics share namespace and task/pipeline label keys — enabling indirect correlation by label matching in a backend like Grafana. However, no trace exemplars on histograms, and logs are not in the OTel pipeline.

Checklist

  • ✓ Two signals (traces + metrics) covering the same operational domain
  • ✓ Metrics cover performance, availability, and scheduling latency
  • ✓ Shared labels enable indirect trace-metric correlation
  • ✗ No OTLP log export — log signal missing from OTel pipeline
  • ✗ No trace exemplars on histogram metrics
  • ✗ Mixed push/pull collection requires separate configuration
  • ✗ OTLP metrics export requires opt-in
Opportunity: Adding trace exemplars to duration histograms would enable one-click "jump from slow metric to the trace" in Grafana/Tempo. Injecting trace_id into stdout JSON log lines (even without full OTLP export) would enable log-trace correlation in most log aggregators.

Audience & Signal Quality
Usefulness, naming quality, and default usability
Level 2 · Native

Span Naming Quality

Span names are logical, domain-meaningful operation labels — not raw Go function names. An operator can understand a trace without reading source code:

  • PipelineRun:Reconciler, TaskRun:Reconciler — clear entry points
  • createTaskRun, createPod, resolvePipelineState — meaningful k8s operations
  • runNextSchedulableTask, finishReconcileUpdateEmitEvents — domain-relevant lifecycle

One exception: durationAndCountMetrics exposes an internal implementation detail (recording metrics inside a span).

What's Working Well

Metrics are enabled by default (Prometheus port 9090). All 11 Tekton metrics have descriptions. Documentation in docs/metrics.md is comprehensive with types, labels, and status.

Checklist

  • ✓ Span names are logical operations, not raw code paths
  • ✓ All metrics have descriptions
  • ✓ Comprehensive metrics documentation
  • ✓ Metrics available by default (no config required)
  • ✗ Traces disabled by default — must opt-in
  • ✗ No step-level spans (most valuable for CI/CD)
  • ✗ durationAndCountMetrics span exposes implementation detail
  • ✗ All metrics marked "experimental" despite production stability
Opportunity: Step-level spans are the single highest-value improvement for a CI/CD system — they would let operators pinpoint slow or failing steps without reading logs. Enabling traces by default (with sampling) would ensure operators have trace data from day one.

Stability & Change Management
Documentation, breaking-change communication, and schema versioning
Level 2 · Native

What We Observed

The docs/metrics-migration-otel.md file is an exemplary piece of breaking-change documentation for the OpenCensus → OpenTelemetry migration. It includes:

  • Before/after tables for every changed metric name
  • Impact levels (HIGH/MEDIUM/LOW) per category
  • Configuration migration steps
  • Dashboard and alert update guidance
  • A FAQ section
  • A quick-reference checklist

Schema URLs are set: traces use 1.12.0, metrics use 1.18.0 — present but stale vs current 1.29.0+.

Checklist

  • ✓ Breaking telemetry changes documented with dedicated migration guide
  • ✓ Metrics documented with types, labels, and status
  • ✓ Schema URL present in trace and metric exports
  • ✓ Configuration options documented inline in ConfigMaps
  • ✗ All metrics marked "experimental" — no stable telemetry contract
  • ✗ Schema URLs are stale (1.12.0 / 1.18.0 vs current 1.29.0+)
  • ✗ No stability documentation for trace span names or attributes
  • ✗ Tracing configuration less documented than metrics
Opportunity: Graduating some metrics from "experimental" to "stable" would give operators confidence to build production dashboards and alerts. Updating schema URLs and adding a tracing documentation page comparable to docs/metrics.md would complete the picture for Level 3.
💡 Key Findings

Strengths

  • Genuine OTel SDK adoption — full migration from OpenCensus to go.opentelemetry.io/otel with working OTLP export
  • Rich reconciler span hierarchy with logical operation names and correct parent-child relationships
  • Innovative spanContext persistence in k8s object status to bridge reconciliation loop boundaries
  • Exemplary breaking-change documentation in metrics-migration-otel.md — a model for the ecosystem
  • 11 well-described Tekton metrics covering duration, throughput, and scheduling latency
  • HTTP auto-instrumentation uses current stable OTel semconv correctly

Priority Improvements

  • Add step-level spans — highest-value improvement for a CI/CD system; pinpoint slow/failing steps without reading logs
  • Adopt OTel semantic conventions for span attributes — propose tekton.* namespace; replace taskrun/namespace keys
  • Support OTEL_* env vars — enable standard operator tooling and OTel Operator compatibility
  • Enable OTLP traces by default (or at low sampling rate) — most deployments currently have zero trace data
  • Add OTLP log export or inject trace_id into stdout JSON for cross-signal correlation
  • Update schema URLs to current version and set instrumentation scope versions
🔬 Notable Observations
⚠️
Two overlapping tracing configuration mechanisms: Both config-tracing (HTTP-only, Jaeger-focused) and config-observability (gRPC/HTTP, newer) exist simultaneously. Only config-observability's tracing-protocol: grpc activates OTLP gRPC. This dual-config is confusing and should be consolidated into a single mechanism.
🔍
Metrics scope identity mismatch: The Prometheus-scraped metrics carry instrumentation scope tekton_pipelines_controller, while traces use pipelinerun-reconciler / taskrun-reconciler as service.name. These don't align, making cross-signal identity matching harder in observability backends.
📌
All metrics are "experimental": Despite being stable, production-used metrics for years, Tekton marks all metrics as "experimental" in docs/metrics.md. This creates unnecessary uncertainty for operators building dashboards and alerts. Graduating core metrics to "stable" would be a low-cost, high-value improvement.
💡
durationAndCountMetrics span: Recording metrics inside a span is an unusual pattern that exposes an internal implementation detail. This span could be removed or renamed to something more user-meaningful (e.g., recordRunMetrics).
